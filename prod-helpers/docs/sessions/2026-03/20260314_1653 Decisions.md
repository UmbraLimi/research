# Session Decisions - 2026-03-14

## 1. Durable Warnings for Conv State Checks in /r-resume
**Type:** Implementation
**Topics:** skills, workflow

**Trigger:** Two open design questions — should /r-resume warn without .conv-current, and how to handle raw /clear mid-conv.

**Options Considered:**
1. Simple — no warning for missing .conv-current; rely on existing /r-start warning for stale state
2. Durable — soft warning for missing .conv-current; stale context detection when .conv-current exists but RESUME-STATE.md is stale/missing ← Chosen

**Decision:** Added two soft warnings to /r-resume SKILL.md: "No active conv" when .conv-current is absent, and "Stale context" when .conv-current exists but RESUME-STATE.md doesn't match. Both are non-blocking.

**Rationale:** Soft warnings give the user information without preventing legitimate use cases (peeking at status, continuing after accidental /clear). The durable approach catches more edge cases without adding friction.

**Consequences:** /r-resume SKILL.md updated with Conv State Checks section and new pre-computed context for .conv-current.

## 2. Delete RESUME-STATE.md When All Items Complete
**Type:** Strategic
**Topics:** skills, workflow

**Trigger:** After consolidation, RESUME-STATE.md had all items checked off but the file still existed, creating noise for future /r-resume and /r-save-state runs.

**Options Considered:**
1. Clean up Remaining — move checked items to Completed, leave Remaining empty
2. Delete the file — RESUME-STATE.md has served its purpose when nothing is pending ← Chosen

**Decision:** Delete RESUME-STATE.md when all remaining items are done. Both /r-resume and /r-save-state auto-detect and auto-delete. State is preserved in git history and PLAN.md.

**Rationale:** RESUME-STATE.md exists to carry unfinished work. A file with only completed items is redundant with git history and adds confusion about whether there's real pending work.

**Consequences:** Added "All-Done Cleanup" section to /r-resume SKILL.md and "All-Done Check" step to /r-save-state SKILL.md.

## 3. Unified Entry Point — Always /r-start
**Type:** Strategic
**Topics:** workflow

**Trigger:** Flowchart had a decision diamond forcing user to choose between /r-start (cold) and /r-resume (warm). User identified this as unnecessary complexity and a risk of forgetting to start a conv.

**Options Considered:**
1. Keep two entry points — user decides based on context
2. Always /r-start — it already calls /r-resume internally, handles both cases ← Chosen

**Decision:** /r-start is the only entry point for all conversations. Warm restart flow changes from `/clear` → `/r-resume` to `/clear` → `/r-start`. /r-resume becomes an internal skill called by /r-start, not a user-facing entry point.

**Rationale:** One habit to build, no decision fatigue, conv counter always incremented. /r-start already handles RESUME-STATE.md via its Step 6 call to /r-resume.

**Consequences:** Updated CONV-FLOWCHART.md (removed decision diamond), CLAUDE.md (unified workflow), r-pre-clear SKILL.md (instructions say /r-start after /clear), r-start SKILL.md (documented as sole entry point).
