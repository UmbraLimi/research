# Conv Learnings - 2026-03-22

## 1. Obsidian CLI v1.12+ Is Production-Ready
**Topics:** obsidian

**Context:** Open question in PLAN.md about whether Obsidian CLI commands are available in v1.12.

**Learning:** Official CLI shipped Feb 27, 2026 in v1.12.4. 100+ commands covering files, search, daily notes, properties, tags, tasks, and developer tools. Key advantage: `move` auto-rewrites wikilinks. Requires Obsidian desktop running. No auth needed (local IPC). This resolves the open question completely — the CLI is the primary interface to target.

## 2. Dynalist Has No Server-Side Search API
**Topics:** dynalist

**Context:** Researching Dynalist API for Phase 1.1 audit.

**Learning:** There is no search endpoint in the Dynalist API. To search, you must fetch the full document via `/doc/read` and filter client-side. Tags are also inline text (`#tag`, `@tag`) in the content field — no dedicated tags API. This means any cross-app search skill will need to handle Dynalist search entirely in-memory.

---

## 3. Joplin Data API Has No Rate Limits
**Topics:** joplin

**Context:** Researching Joplin API for Phase 1.1 audit.

**Learning:** The Joplin Data API runs on localhost with no rate limiting. Default port 41184 (scan 41184-41194 if unavailable). It has proper tag CRUD (unlike Dynalist), search endpoint, and events API for change tracking. The main constraint is Joplin desktop must be running.

## 4. "As-Designed" vs "As-Built" Resolves Architecture Naming
**Topics:** docs-infra

**Context:** Trying to rename `architecture/` to `design/` for a unified docs structure across coding and non-coding projects.

**Learning:** The user drew a crucial distinction: "design is for what's being designed, architecture is what got built." This led to `as-designed/` (pre-build specs) and `as-built/` (post-build docs) — two folders with a clear lifecycle relationship. The `as-` prefix makes the temporal distinction unmistakable. Generic names like "design" or "architecture" are ambiguous about whether they describe intent or reality.

## 5. Decision Routing Requires Active Enforcement
**Topics:** docs-infra, workflow

**Context:** After establishing dual routing (DECISIONS.md for project domain, DOC-DECISIONS.md for workflow), reviewing existing entries.

**Learning:** When DECISIONS.md was first populated (Conv 003-007), the DOC-DECISIONS.md split didn't exist yet. Result: 15 of 17 Tooling & APIs entries were actually workflow/convention decisions. Lesson: when introducing a new organizational boundary, audit existing content against the new criteria — don't assume past entries are correctly placed.
