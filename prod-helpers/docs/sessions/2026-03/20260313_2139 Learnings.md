# Session Learnings - 2026-03-13

## 1. Skills 2 Format Should Be Preserved When Adapting Skills Between Projects
**Topics:** skills, workflow

**Context:** Adapted 7 skills from peerloop-docs (Skills 2 format with `allowed-tools` and `!` interpolation) to prod-helpers. Initially converted them to the older `.claude/commands/` format to match the existing `/r-commit` file, losing Skills 2 features.

**Learning:** When adapting skills from a project that uses Skills 2 format, keep them in Skills 2 format. The features (`allowed-tools`, `!` shell interpolation, `${CLAUDE_SKILL_DIR}` paths) exist for good reasons. Matching an older file's format is not worth losing those capabilities.

**Pattern:** Skills 2 structure: `.claude/skills/{skill-name}/SKILL.md` with frontmatter containing `name`, `description`, `argument-hint`, `allowed-tools`.

## 2. Peerloop-docs Documentation Structure Is a Proven Pattern Worth Reusing
**Topics:** workflow, skills

**Context:** Explored three existing projects (peerloop-docs, translation-system, Comm-Helpers) to find organizational patterns for a new project.

**Learning:** The `docs/` folder structure from peerloop-docs (`sessions/`, `reference/`, `architecture/`, `vendors/`, `guides/`) provides good separation of concerns. `sessions/` for ephemeral logs, `reference/` for API docs, `architecture/` for design decisions — each has a clear purpose and update trigger.

---

## 3. Planning Projects Don't Need Code-Level Change Detection Scripts
**Topics:** skills, workflow

**Context:** The peerloop-docs `/w-docs` skill has 4 helper scripts for automated change detection (`detect-changes.sh`, `sync-gaps.sh`, `tech-doc-sweep.sh`, `dev-env-scan.sh`). Considered whether to port these to prod-helpers.

**Learning:** A planning/research project without a codebase doesn't need automated change detection. Simple `git diff` and `find` commands in `!` interpolation are sufficient. The scripts can be added later if the project grows to need them.

## 4. Ancillary Files in Skills 2 Directories Serve Specific Purposes
**Topics:** skills

**Context:** Compared file listings between peerloop-docs `.claude/skills/` and the newly created prod-helpers skills. Found that `w-docs/` had `scripts/`, `.last-qdocs-run`, and `.sync` files beyond just SKILL.md.

**Learning:** Skills 2 directories can contain supporting files (scripts, data files, state markers) referenced via `${CLAUDE_SKILL_DIR}`. When adapting skills, check for these ancillary files — they may be essential to the skill's function or may be project-specific artifacts that don't need porting.

---

## 5. JavaScript/Node Preferred Over Python for Script Rigor
**Topics:** workflow, skills

**Context:** Discussed whether scripts in this project should use Python or JavaScript. Python is common in Claude Code skills (e.g., `gmail-thread.py` in Comm-Helpers) but those cases were driven by library dependencies.

**Learning:** Default to JavaScript/Node with vitest for scripts unless a specific library dependency requires Python. The user prefers JS rigor and none of the planned integrations (Dynalist REST API, Joplin REST API, Obsidian CLI, Whisper CLI) require Python.
