# Session Learnings - 2026-03-14

## 1. Skills vs Commands — Technical Requirements Override Consistency
**Topics:** skills, workflow

**Context:** Needed to create r-commit with `!` backtick pre-computation for machine name. Placed it in `.claude/commands/` to match sibling projects (Comm-Helpers, Alpha-Peer, etc.).

**Learning:** `!` backtick pre-computation only works in `.claude/skills/` (SKILL.md format), not `.claude/commands/`. When a feature requires skills-only capabilities, the file must go in skills — organizational consistency with other projects cannot override technical correctness. When there's a tension between the two, surface the conflict to the user rather than silently picking one side.

## 2. Conversation vs Session — Naming Matters for Tooling
**Topics:** workflow, skills

**Context:** Designing cross-machine session numbering. Initial options (git-committed counter, git tags, derived from history) all had "forgot to sync" failure modes.

**Learning:** "Session" is ambiguous — it could mean a Claude Code conversation (one invocation) or a unit of work (one commit). A conversation can produce multiple commits, and session docs (dumps, learnings) naturally span the full conversation. Using "conv" (short for conversation) eliminates the ambiguity. The auto-increment tracks conversations, and multiple commits within one conversation share the same conv number.

## 3. Bulletproof Cross-Machine Sync Requires Automated Sync Points
**Topics:** workflow, skills

**Context:** Two machines (MacMiniM4, MacMiniM4-Pro) share the same git repo, never simultaneous. Needed a conversation counter that stays in sync.

**Learning:** Any sync mechanism that depends on human memory (remembering to pull/push) is fragile. The bulletproof approach: auto-pull at conversation start + immediate commit-and-push of the counter, before any work begins. This removes the human from the sync loop entirely. `/r-start` enforces pull→increment→push; `/r-end` enforces commit→push→cleanup.

**Pattern:** Counter file (`CONV-COUNTER`) committed to git + ephemeral lock file (`.conv-current`, gitignored) for the active session. The lock file protects against mid-session drift and signals unclean exits.
