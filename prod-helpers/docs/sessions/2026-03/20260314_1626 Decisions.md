# Session Decisions - 2026-03-14

## 1. RESUME-STATE.md append ordering: oldest first
**Type:** Implementation
**Topics:** skills, workflow

**Trigger:** Adding append mode to `/r-save-state` — needed to decide block ordering in the file.

**Options Considered:**
1. Option A: Simple concat, newest at bottom (oldest first) ← Chosen
2. Option B: Merged sections (combine old+new into one unified block)
3. Option C: Newest first (latest block at top)

**Decision:** Append new state blocks below existing content, oldest first. Conv number and date/time in each block header confirm chronological order.

**Rationale:** Oldest-first is the natural reading order and matches how the file grows over time. Conv labels make the order self-evident. Option B was rejected because merging at save time is premature — the consolidation belongs in `/r-resume` where the model can evaluate what's actually done. Option C would require `/r-resume` to skip to the top, adding unnecessary complexity.

**Consequences:** `/r-resume` reads the last block as authoritative; earlier blocks are historical context for consolidation.

## 2. Max 2 blocks with /r-resume as the resolution point
**Type:** Strategic
**Topics:** skills, workflow

**Trigger:** Append mode without a cap would create unbounded block growth across warm restarts.

**Options Considered:**
1. No cap — let blocks accumulate, `/r-resume` handles any number ← Rejected
2. Max 2 blocks — `/r-resume` consolidates before normal resume, `/r-save-state` refuses 3rd append ← Chosen

**Decision:** RESUME-STATE.md holds at most 2 state blocks. `/r-save-state` refuses to append if 2 already exist (tells user to run `/r-resume` first). `/r-resume` consolidates multi-block files back to 1 block before presenting resume context.

**Rationale:** Keeps the file manageable and the consolidation logic simple (always 2→1, never N→1). The closed loop guarantees the file resets each cycle.

**Consequences:** `/r-save-state` checks block count before appending. `/r-resume` gained Write, Edit, and Bash permissions to perform consolidation and rewrite the file.

## 3. Consolidation requires evidence-based explanation before rewriting
**Type:** Strategic
**Topics:** skills, workflow, agents

**Trigger:** Initial consolidation design showed groupings and asked "rewrite? y/n" — user wanted to see the reasoning, not just the result.

**Options Considered:**
1. Silent rewrite — consolidate and rewrite without asking ← Rejected
2. Summary groupings with y/n confirmation ← Rejected (initial design)
3. Full evidence-based explanation per item, then wait for approval ← Chosen

**Decision:** `/r-resume` must explain every classification: "done because [evidence]", "pending because [evidence]", "interacts because [relationship]", "dropping because [reason]". Wait for user approval before rewriting.

**Rationale:** The model may misjudge whether work is complete or misread interactions between blocks. Showing evidence lets the user catch mistakes before the file is rewritten. Aligns with the project's "ask before deciding" principle.

**Consequences:** Updated `/r-resume` SKILL.md consolidation Step B with evidence-based format. Updated skills-overview.md interaction model.
