# Conv Learnings - 2026-03-21

## 1. $CLAUDE_PROJECT_DIR Not Available in `!` Backtick Pre-computation
**Topics:** cc-workflow, skills

**Context:** After bulk-updating all 10 r-* skills to use `$CLAUDE_PROJECT_DIR` paths (ported from peerloop-docs), `/r-end` immediately failed — the env var expanded to empty in `!` backtick commands.

**Learning:** `$CLAUDE_PROJECT_DIR` is NOT available during `!` backtick pre-computation in SKILL.md files. The `!` backtick execution environment does not inherit the same env vars that hooks receive. The correct pattern is: use **relative paths** in `!` backtick lines (`.claude/scripts/foo.sh`), and have the **scripts themselves** resolve paths internally via `${CLAUDE_PROJECT_DIR:-$(pwd)}`.

**Pattern:**
- SKILL.md `!` backtick line: `!`.claude/scripts/foo.sh``
- Inside foo.sh: `DOCS_ROOT="${CLAUDE_PROJECT_DIR:-$(pwd)}"`

> This is the same learning documented in peerloop-docs Conv 003 (20260317_2019 Learnings.md). It re-surfaced because the SKILL.md-level pattern from peerloop-docs was blindly ported without recognizing the difference: peerloop-docs may have a different runtime environment or the issue was addressed there by other means.

## 2. Cross-Project Skill Porting Requires Environment Awareness
**Topics:** cc-workflow, skills, docs-infra

**Context:** Bulk comparison of r-* skills between prod-helpers and peerloop-docs revealed many improvements in peerloop-docs (dirty-repo guard, structured commits, insight capture, TodoWrite-all-gaps rule, etc.). Direct porting was attempted.

**Learning:** When porting skills between projects, the structural/behavioral improvements can be taken directly, but anything touching runtime environment (`!` backtick paths, `git -C` patterns for multi-repo, `$CLAUDE_SKILL_DIR` references) must be adapted to the target project's environment. peerloop-docs is a dual-repo setup; prod-helpers is single-repo within a monorepo. The `!` backtick path strategy is the most critical difference.

## 3. Dual-Repo vs Single-Repo Skill Differences Are Structural
**Topics:** cc-workflow, docs-infra

**Context:** peerloop-docs r-* skills manage two git repos (docs + code) with `git -C` commands everywhere. prod-helpers manages one folder within a monorepo.

**Learning:** The following peerloop-docs patterns should NOT be ported to prod-helpers: dual `git -C` pull/push, dual dirty-repo checks, dual commit reporting, `Branch: code: [x], docs: [x]` in save-state. The single-repo equivalents are simpler: plain `git` commands scoped to the current folder via `git add .` and `git status -- .`.
