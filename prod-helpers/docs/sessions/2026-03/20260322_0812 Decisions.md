# Conv Decisions - 2026-03-22

## 1. Machine-Specific Credential Storage
**Type:** Implementation
**Topics:** credentials, workflow

**Trigger:** Need to store API tokens for Dynalist and Joplin, different per machine.

**Options Considered:**
1. Single `.env.<machine>` file per machine
2. `credentials/` directory with one file per machine ← Chosen
3. Single YAML file with all machines

**Decision:** Use `credentials/` directory with `MacMiniM4.env`, `MacMiniM4-Pro.env` etc. Entire directory gitignored.

**Rationale:** Cleanest separation. No merge conflicts since file is untracked. Each machine edits only its own file.

**Consequences:** Created `credentials/MacMiniM4.env` with empty token fields. Added `credentials/` to `.gitignore`.

## 2. Unified Docs Folder Structure (6 Standard Folders)
**Type:** Strategic
**Topics:** docs-infra

**Trigger:** Four projects evolved different docs/ structures. Comm-Helpers merging into prod-helpers required standardization.

**Options Considered:**
1. Keep project-specific structures
2. Standardize a common set of folders ← Chosen

**Decision:** Six standard folders: `sessions/`, `requirements/`, `reference/`, `as-designed/`, `as-built/`, `guides/`. Lifecycle: requirements → as-designed → build → as-built. Folders created as needed, never pre-created empty.

**Rationale:** Works for both coding projects (peerloop) and non-coding projects (prod-helpers, Comm-Helpers). The as-designed/as-built distinction captures the temporal relationship between specs and implementations.

**Consequences:** Applied to prod-helpers, documented in ~/research/CLAUDE.md and Minimum-Project/README.md. Peerloop migration deferred to future conv.

## 3. Rename architecture/ → as-designed/ + Add as-built/
**Type:** Strategic
**Topics:** docs-infra

**Trigger:** Generic "architecture" name was ambiguous — could mean pre-build design or post-build documentation.

**Decision:** `as-designed/` for pre-build specs (tag-mapping.md, note-formats.md). `as-built/` for post-build documentation of implemented systems. The `as-` prefix makes the distinction unmistakable.

**Rationale:** User insight: "design is for what's being designed, architecture is what got built. I think we need both."

**Consequences:** Moved 3 files from architecture/ to as-designed/. Updated 3 skills and CLAUDE.md. Peerloop's architecture/ will become as-built/ (those are post-build docs).

## 6. RFCs and User Stories in docs/requirements/
**Type:** Strategic
**Topics:** docs-infra

**Trigger:** RFCs (peerloop) and user stories are common across coding and non-coding projects. Need a standard home.

**Decision:** `docs/requirements/` with flat conventions: `user-stories.md` (or `user-stories/` dir), `rfcs/` subfolder with `INDEX.md` and numbered `RFC-001/` folders. Processed outputs only — no raw source material in the standard.

**Rationale:** Requirements are inputs to design, sitting upstream of as-designed/. They belong in docs/ alongside the rest of the documentation lifecycle.

**Consequences:** Added to standard. No folders created yet in prod-helpers (will be created when content exists).
