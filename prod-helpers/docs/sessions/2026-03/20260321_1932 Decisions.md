# Session Decisions - 2026-03-21

## 1. Per-project conv numbering with 3-letter prefix
**Type:** Strategic
**Topics:** workflow

**Trigger:** Conv numbering lives in prod-helpers but git ops affect the entire research/ repo. A second project with its own counter would collide.

**Options Considered:**
1. One shared counter at repo root — simple but gaps per project
2. Per-project counters with project prefix ← Chosen
3. Keep as-is, only one project uses it — breaks when second project added

**Decision:** Each project gets its own CONV-COUNTER with a 3-letter prefix (e.g. PRH-004). Commit messages include the folder name for scannability.

**Rationale:** Sequential per-project numbering is more useful than a shared counter with gaps. The prefix registry prevents collisions.

**Consequences:** /r-start, /r-end, /r-commit skills need updating to read PROJECT.yaml and format conv numbers with prefix.

## 2. YAML project identity files with repo-level registry
**Type:** Implementation
**Topics:** workflow

**Trigger:** Need to store the 3-letter prefix per project, and anticipate more project metadata fields.

**Options Considered:**
1. Plain text files (PREFIX only) — simple but no room to grow
2. YAML with structured fields ← Chosen

**Decision:** Each subfolder gets `PROJECT.yaml` (with `prefix:` field). Repo root gets `PROJECTS.yaml` as the authoritative registry keyed by prefix.

**Rationale:** YAML accommodates future fields without reformatting. The root file prevents prefix collisions; the subfolder file keeps skills self-contained.

**Consequences:** Minimum-Project template includes PROJECT.yaml with placeholder `XXX`. New projects must register in PROJECTS.yaml.
