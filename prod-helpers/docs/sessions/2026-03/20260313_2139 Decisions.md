# Session Decisions - 2026-03-13

## 1. Adopt Peerloop-docs Documentation Folder Structure
**Type:** Strategic
**Topics:** workflow

**Trigger:** Needed to organize documentation for a new project. User asked to examine other projects for patterns worth adopting.

**Options Considered:**
1. Flat docs folder
2. Peerloop-docs structure (sessions/, reference/, architecture/, vendors/, guides/) ← Chosen
3. Custom structure

**Decision:** Use the peerloop-docs `docs/` folder hierarchy: `sessions/` for session logs, `reference/` for API docs, `architecture/` for design docs, `vendors/` for external tool notes, `guides/` for how-to docs.

**Rationale:** Proven across 386+ sessions in peerloop-docs. Clear separation by purpose and update frequency.

**Consequences:** Created skeleton reference and architecture docs aligned to PLAN.md phases.

## 2. Use Skills 2 Format for All Project Skills
**Type:** Implementation
**Topics:** skills

**Trigger:** Initially placed adapted skills in `.claude/commands/` (old format). User questioned why Skills 2 features were dropped.

**Options Considered:**
1. `.claude/commands/*.md` (old format, matches existing r-commit)
2. `.claude/skills/{name}/SKILL.md` (Skills 2 format) ← Chosen

**Decision:** All project skills use Skills 2 format in `.claude/skills/r-*/SKILL.md`.

**Rationale:** Preserves `allowed-tools` restrictions and `!` shell interpolation from the source skills. These features are valuable — no reason to downgrade.

**Consequences:** Migrated all 7 skills. Removed old `.claude/commands/` files.

## 3. JavaScript/Node as Default Script Language
**Type:** Strategic
**Topics:** workflow, skills

**Trigger:** Discussion about whether scripts should use Python or JavaScript.

**Options Considered:**
1. Python (common in Claude Code skills, Whisper SDK)
2. JavaScript/Node with vitest ← Chosen
3. Mix based on need

**Decision:** Default to JavaScript/Node for scripts. Only use Python when a specific library dependency requires it.

**Rationale:** User prefers JS rigor. No planned integrations require Python — all target APIs are REST-based, Whisper can be invoked as CLI.

**Consequences:** Will set up `package.json` + vitest when the first script is needed (not pre-emptively).

## 4. Defer Script Infrastructure Until First Script Is Needed
**Type:** Strategic
**Topics:** workflow

**Trigger:** Discussion about whether to set up package.json and testing now.

**Options Considered:**
1. Set up package.json + vitest now
2. Wait until the first script is actually needed ← Chosen

**Decision:** No premature infrastructure. Establish script organization structure when the first script is created.

**Rationale:** Avoids over-engineering. Structure decisions are better made with a concrete use case in hand.

**Consequences:** When the first script is needed, must design directory structure, naming conventions, and testing approach before writing code.

## 5. Simplified r-docs Skill Without Detection Scripts
**Type:** Implementation
**Topics:** skills

**Trigger:** The peerloop-docs `/w-docs` skill has 4 helper bash scripts for automated change detection. Needed to decide whether to port them.

**Options Considered:**
1. Port all 4 detection scripts adapted for prod-helpers
2. Use simple git diff + find in `!` interpolation ← Chosen
3. No automated detection at all

**Decision:** Use simple pre-computed context (git diff, find) instead of dedicated detection scripts.

**Rationale:** A planning/research project doesn't have the code surface area that justifies automated detection. Scripts can be added later if needed.

**Consequences:** r-docs relies on session context and simple git queries rather than marker-based change tracking.
