# Session Decisions - 2026-03-14

## 1. Adopt Machine Tracking in Commit Messages
**Type:** Implementation
**Topics:** workflow, skills

**Trigger:** Peerloop-docs project already records which machine work was done on; user wanted the same in prod-helpers.

**Options Considered:**
1. Add `Machine:` line to commit messages via `.claude/commands/` (matching sibling projects)
2. Add `Machine:` line via `.claude/skills/` with `!` backtick pre-computation ← Chosen

**Decision:** Machine name is pre-computed via `!` backtick reading `~/.claude/.machine-name` and included in every commit message body as `Machine: {value}`.

**Rationale:** The `!` backtick pre-computation requires skills format, and deterministic injection is a core project principle.

**Consequences:** Created `.claude/skills/r-commit/SKILL.md` with machine pre-computation. Existing sibling projects still use commands format — user will upgrade those separately if needed.

## 2. Use "Conv" as Short-Form for Conversation Numbering
**Type:** Strategic
**Topics:** workflow, skills

**Trigger:** Need for human-friendly, auto-incrementing identifiers across two machines that gives a sense of recency/distance.

**Options Considered:**
1. "Session" — already established in peerloop-docs but ambiguous (conversation vs commit)
2. "Conversation" — precise but verbose
3. "Conv" — short, unambiguous, obvious ← Chosen

**Decision:** Use "conv" everywhere: `Conv: 014` in commits, `conv-014` in filenames, "Conv 14" in speech. Zero-padded to 3 digits.

**Rationale:** Eliminates the session/commit ambiguity. Multiple commits in one conversation share the same conv number, which is the correct semantic.

**Consequences:** New files: `CONV-COUNTER`, `.conv-current`. New skills: `/r-start`, `/r-end`. Updated `/r-commit` to include `Conv:` line.

## 3. Auto-Synced Counter with Mandatory Pull/Push
**Type:** Strategic
**Topics:** workflow, skills

**Trigger:** Two machines (MacMiniM4, MacMiniM4-Pro) need a conversation counter that never drifts.

**Options Considered:**
1. Git-committed counter file with manual sync — "forgot to push" risk
2. Derive from git history — sessions with no commits get no number
3. Git tags — requires explicit `git push --tags`
4. Counter file + hybrid fallback — extra complexity
5. Auto-synced counter file via `/r-start` and `/r-end` ← Chosen

**Decision:** `CONV-COUNTER` is committed to git. `/r-start` auto-pulls, increments, commits, and pushes before any work begins. `/r-end` commits all work and pushes. Both skills HALT on sync failure.

**Rationale:** Removes human from the sync loop entirely. Every other option had a "forgot to sync" failure mode.

**Consequences:** `/r-start` replaces `/r-resume` as the conversation entry point. `/r-end` replaces `/r-eos` + manual `/r-commit`. Both enforce git sync as mandatory steps.

## 4. Separate /r-start and /r-end Skills
**Type:** Implementation
**Topics:** skills

**Trigger:** Needed a mechanism to open and close conversations with counter management.

**Options Considered:**
1. Single combined skill with start/end mode switching
2. Separate `/r-start` and `/r-end` skills ← Chosen

**Decision:** Two distinct skills. `/r-start` handles pull→increment→push→resume. `/r-end` handles eos→commit→push→cleanup.

**Rationale:** Start and end have completely different responsibilities and tool needs. Combining them would require complex branching logic with no benefit.

**Consequences:** `/r-start` wraps `/r-resume`; `/r-end` wraps `/r-eos` + `/r-commit`. The wrapped skills remain available as building blocks.
