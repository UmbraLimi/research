# Session Learnings - 2026-03-14

## 1. Claude Code Permission Checker Treats Pipes as Multi-Operation Commands
**Topics:** skills, workflow

**Context:** The `/r-resume` skill failed on `!` backtick pre-computed commands containing shell pipes (`|`). The permission checker rejected `sed ... | head` as "multiple operations."

**Learning:** The permission checker matches the full command string as a single unit — it does not split on pipes and check each segment against allow patterns independently. `Bash(sed:*)` allows `sed <anything>` but not `sed ... | head ...` because the pipe makes the full string fail the prefix match.

**Pattern:** Wrapping piped commands in shell scripts (`.claude/scripts/`) lets the `!` backtick call a single command while the pipe logic lives inside the script. One blanket allow rule — `Bash(.claude/scripts/*)` — covers all scripts.

## 2. Settings.local.json Changes Require a Session Restart
**Topics:** skills, workflow

**Context:** After adding `Bash(sed:*)` and other broad patterns to `settings.local.json`, the skill still failed on the next invocation. It worked on a subsequent attempt later in the session.

**Learning:** Permission settings changes are not guaranteed to take effect immediately in the current session. When debugging permission failures after a settings change, a session restart may be needed before the patterns are picked up.

---

## 3. The `:*` vs `*` Wildcard Syntax in Permission Patterns
**Topics:** skills, workflow

**Context:** The pattern `Bash(.claude/scripts/:*)` failed to match `.claude/scripts/plan-status-header.sh`. Changing it to `Bash(.claude/scripts/*)` worked.

**Learning:** The `:*` syntax expects a command name followed by arguments (space-separated). For path-based commands with no arguments, plain `*` glob works. Use `:*` for `command <args>` patterns, `*` for path prefix matching.

---

## 4. `!` Backtick Pre-Computed Context is a Design-Time Determinism Feature
**Topics:** skills

**Context:** Discussed whether to replace `!` backtick commands with tool-based alternatives (Read, Grep) when facing permission issues.

**Learning:** Pre-computed context shifts responsibility from Claude (runtime decisions) to the skill author (design-time guarantees). It eliminates tool-call round-trips, reduces token cost on "what should I gather?" reasoning, and enables coordinated context (e.g., shared timestamps across sub-skills). Replacing `!` backticks with tool calls loses this determinism — fix the permission or the command instead.

---

## 5. Wrapper Scripts Provide Self-Documenting Skill Readability
**Topics:** skills, workflow

**Context:** After confirming that broad `:*` patterns make inline pipes work, the user chose to keep the wrapper scripts anyway.

**Learning:** Script filenames like `plan-status-header.sh` and `decisions-categories.sh` make skills more readable than inline pipes like `sed -n '1,/^## Phase/p' PLAN.md 2>/dev/null | head -20`. The indirection preserves `!` backtick build-time execution while adding self-documentation. This pattern is already used in peerloop-docs (`w-docs` skill with `${CLAUDE_SKILL_DIR}/scripts/`).
